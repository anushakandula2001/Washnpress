1. Purpose
This document specifies the Wash N Press mobile applications at screen level: screen-by-
screen field definitions, validations, states, and user flows. It is intended for use directly by
UI/UX designers and engineers building the Resident App and the Operations App, and should
be read alongside the platform-level Functional Requirements Document and DPR for business
context.
Two applications are covered: the Resident App (consumer-facing, iOS/Android) and the
Operations App (used by women's-unit operators and, where applicable, hub-model
pickup/delivery agents and facility workers).
2. Resident App — Screen-Level Specification
2.1 Screen Flow Overview
• Splash → Location/Society Selection → Sign Up/Login (OTP) → Profile Setup → Home
• Home → Plans → Plan Detail → Payment → Subscription Active
• Home → Schedule Pickup → Slot Selection → Confirmation
• Home → Order Tracking → Order Detail → Rate/Dispute
• Home → Wallet/Billing, Support, Profile/Settings (tab navigation)

3. Operations App — Screen-Level Specification
The Operations App serves two primary modes: Unit Mode (default, for women's-unit operators
working within a single society) and Route Mode (for hub-model pickup/delivery agents and
facility workers, used only in high-density clusters operating the centralized hub model). Login
determines which mode loads.
3.1 Login & Daily Start
9. Login via mobile number + OTP or staff ID + PIN (configurable by deployment).
10. Unit Mode: lands on "Today's Bookings" screen scoped to the operator's society.
11. Route Mode: lands on "Today's Route" screen showing the assigned stop sequence.
3.2 Unit Mode — Today's Bookings
Element Detail
Booking list Grouped by time slot; each entry shows resident name (masked per privacy
settings), flat number, pickup/delivery type
Status actions Per booking: "Mark Picked Up" / "Mark Delivered" buttons
Garment logging On "Mark Picked Up": opens garment entry screen (category + count fields,
e.g., Shirts: 4, Trousers: 2, Bedsheets: 1)
QR tagging After garment entry, app generates/prints or displays a QR batch code to
physically tag the garment bag
3.3 Unit Mode — Processing Pipeline
12. Scan QR on batch arrival at the in-society processing point → status moves to "In
Wash."
13. Operator manually advances status through Wash → Dry → Iron, each a single-tap
action with optional timer (using the recipe-style timer pattern for wash/dry cycles).
14. QC screen: pass/fail toggle per batch; if Fail, mandatory reason dropdown (stain not
removed, damage, fabric issue, other) + photo upload — this auto-creates a resident
notification and a support ticket; batch cannot proceed to "Ready for Delivery" until QC is
marked Pass or a documented exception override is approved by Admin.
15. Water usage entry (manual for MVP): liters used per cycle, or auto-populated if
integrated machinery telemetry is available — feeds the Admin sustainability dashboard.
3.4 Unit Mode — Delivery
16. From "Ready for Delivery" queue, operator selects batch, scans QR to pull up the
original pickup manifest.
17. App displays manifest count side-by-side with a re-count entry field — if counts
mismatch, "Mark Delivered" is blocked and a discrepancy reason is required, auto-
generating a support ticket.
18. Delivery confirmation: OTP entered by resident, or digital signature capture.

3.5 Route Mode — Today's Route (Hub Model)
• Optimized stop list grouped by building/tower with turn-by-turn navigation link per stop.
• Same garment-logging and QR-tagging flow as Unit Mode, applied per stop.
• Route completion summary: total stops, total garments collected, any flagged issues.
3.6 Hub Worker Mode (Hub Model)
• Incoming manifest queue: list of batches en route from all collection points.
• Scan-to-receive at hub; processing pipeline screens identical in structure to Unit Mode
(Wash → Dry → Iron → QC).
• Outgoing dispatch queue: batches ready for the delivery route, grouped by route/agent.
3.7 Earnings & Performance (Unit Mode)
Element Detail
Earnings summary Current cycle: base draw amount, orders processed, performance-linked
share accrued, projected payout
Performance metrics Quality score (QC pass rate), on-time pickup/delivery rate, complaint count
— visible to operator for transparency
Training access Library of training videos/modules; refresher module completion tracking
Escalation "Report an Issue" — equipment malfunction, supply shortage, safety
concern — routed directly to Ops/Impact Manager
3.8 Offline Behavior
• All pickup/delivery/processing actions are cached locally if connectivity is unavailable
and synced automatically on reconnect.
• Visual indicator (banner) when operating offline; queued actions shown with a pending-
sync icon until confirmed synced.

5. Validation & Error-State Summary
• All monetary fields: non-negative, two-decimal precision, currency-locked to INR.
• All phone number fields: 10-digit Indian mobile format, validated on blur.
• Garment count fields: non-negative integers only; zero allowed (e.g., add-on-only order
with no standard garments).
• QR scan failures (damaged/unreadable code): manual batch-ID entry fallback, logged
with a flag for Admin review.
• Network failure during payment: transaction status polled and reconciled before showing
success/failure to resident — never assume success on timeout.
6. Open Items
• Final decision on masked vs. full resident contact visibility to operators (privacy vs.
operational convenience trade-off).
• Whether QC override (proceeding despite a Fail) requires real-time Admin approval or
can be operator-initiated with after-the-fact audit.
• Granularity of push notification settings exposed to residents (per-stage toggle vs. single
on/off).
• Machinery telemetry integration timeline for automated water-usage logging vs. manual
entry at MVP.
